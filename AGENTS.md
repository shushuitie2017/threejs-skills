# Agent 指令

本仓库包含用于 Three.js 浏览器游戏开发的 Codex 工作流资产。

## 默认技术栈

- 优先使用 TypeScript、Vite、npm 包导入以及 Three.js 模块。
- 使用 `three/addons/...` 获取官方的控制器、加载器和后处理辅助工具。
- 街机式触发器和原型规模的交互使用简单的自定义碰撞。
- 对于带刚体、传感器、球体、斜坡、移动平台、大量接触点或高速碰撞的正式 Three.js 浏览器游戏，将 Rapier 作为默认的稳健物理引擎。
- 仅当需要避免 WASM 时，才把 `cannon-es` 当作小型/简单刚体场景的轻量级 JavaScript 回退方案。
- 在新增或修改重物理玩法之前，先加载 `threejs-gameplay-systems/references/physics-engine-selection.md`。
- 当本地调参能显著加快迭代时，使用 `lil-gui`。
- 当性能纳入考量范围时，使用轻量级 HUD 或 `stats.js` 做帧诊断。
- 把 WebGPU 视为有条件采用。仅当项目能从中受益时才使用 Three.js 的 `WebGPURenderer`，并保留一条 WebGL/WebGL2 回退路径。

## 游戏质量标准

- 对于"构建、升级、打磨或完成一款 Three.js 游戏"这类宽泛请求，先经由 `threejs-game-director` 路由。用户不应被要求逐一点名每个专项技能。
- 公开的 Three.js 技能围绕总监技能加各专项系统整合：玩法系统、AAA 图形、UI、调试/性能剖析、QA/发布、3D 生成、图像生成和音频生成。
- 在 Claude 式的技能运行器中，不要假设总监技能能直接调用其他技能。总监必须先尝试加载同级的公开 `SKILL.md` 文件，报告一份技能加载台账，并且仅对无法加载的文件才使用内置的分阶段 OS。
- 对于宽泛/高端的总监工作，总监必须在进入每个阶段时加载该阶段所需的 `references/*.md` 文件，并报告一份引用台账。若某阶段所需的引用被跳过，则该阶段未完成。
- 高端/AAA/展示级的宣称必须附上 `threejs-aaa-graphics-builder/references/visual-scorecard.md` 中填好的视觉评分卡，包括平均分和尚存的自动判负项。不要用临时拼凑的评分标准替代。
- 当有 shell 工具可用时，在宣称高端、AAA、展示级、完成、可发布或"不那么基础"的成功之前，先针对最终证据报告运行 `skills/threejs-game-director/scripts/audit_reference_report.py --premium <report.md>`。
- 先构建一个可玩循环。静态场景不算完成。
- 当用户要求高端、AAA、打磨、完成、可发布或展示级质量时，不要止步于第一个可玩切片。
- 一旦原型规模超出单个简单文件，就把场景搭建、循环、输入、系统、实体、UI、资产和调试工具分离开来。
- 通过短促的试玩循环来调校移动、相机、碰撞、反馈和 HUD。
- 对于重物理的游戏，报告引擎选择、固定时间步、碰撞体策略、传感器、CCD 使用、刚体/碰撞体数量以及重启清理证据。
- 避免纯装饰性 UI、通用的紫色渐变、粒子堆砌以及未加节制的后处理。
- 把移动端输入和缩放行为放在首次实现路径中，而非最后的补救环节。
- 当视觉优先级不明确时，使用基于截图的美术方向评审。
- 当截图在多个视觉层面上仍显基础时，使用 `threejs-aaa-graphics-builder`。不要把这个宽泛的图形问题拆成一个个孤立的小打磨任务。
- 对 HUD、菜单、文本适配、图标控件和安全区布局使用聚焦的 UI 处理，而不是把界面工艺当作通用打磨。
- 对手工搭建的资产、道具套件、材质变化、实例化、LOD 以及感知渲染器数量的细节，使用聚焦的程序化模型处理。
- 对于涉及角色、载具、飞船、武器、建筑、标志性道具、天空、纹理、贴花、logo、图标、GUI 美术、音效、环境声或人声的高端/AAA/展示级/高保真/不那么基础的工作，在断定不需要程序化/生成资产之前，先加载 `threejs-3d-generator`、`threejs-image-generator` 和/或 `threejs-audio-generator`。
- 在宣称 3D/图像/音频生成凭据不可用之前，先运行总监的凭据探测。除非报告了 SET/MISSING 探测输出，否则缺失密钥不构成有效的阻塞理由。
- 为高端图形工作报告一份外部资产采购台账：每个高价值层面标注 程序化 / threejs-image-generator / threejs-3d-generator / 混合，并附任务 ID/输出路径或真实的阻塞证据。
- 对于高端主视觉层面，仅靠程序化生成不是有效的最终答案，除非凭据探测输出或尝试生成显示出真实的阻塞。
- 对高价值的外部 AI 生成 3D 资产、自动绑骨、动画、纹理、转换以及 GLB/FBX 游戏资产工作流，使用 `threejs-3d-generator`。
- 当 2D 概念/参考图、T-pose/A-pose 角色设定图、纹理参考、贴花、logo、图标、天空、背景或 GUI 美术能改善结果时，先用 `threejs-image-generator` 再用 `threejs-3d-generator`。
- 对游戏音效、环境声循环、UI 音效、播报/对白、人声转换和音频清理，使用 `threejs-audio-generator`。
- 把通用的数据卡片式 HUD、立方体障碍物和天际线方块视为原型占位符，除非用户明确想要那种风格。
- 高端/AAA 宣称需要真实游玩过程的截图评分。若视觉评分卡有任一类别低于 2 分，则该构建不算高端。

## 验证标准

对于有实质意义的 Three.js 改动，在宣称成功之前先收集证据：

- `npm run build`
- 在浏览器中打开本地 dev 或 preview 服务器
- 浏览器控制台和页面错误检查
- Playwright 截图
- canvas 非空白像素检查
- 至少一个桌面视口和一个移动视口
- 主控制路径的交互检查
- 当 draw call、资产数量、着色器或后处理发生变化时的性能快照
- 当界面布局发生变化时，提供 UI 文本适配、重叠、安全区和触摸目标证据
- 当程序化模型保真度、材质数量或重复道具发生变化时的渲染器诊断
- 当高端图形包含高价值 3D 或 2D 资产类别时的外部资产采购台账
- 针对高端资产类别宣称，提供凭据探测输出加上真实的外部资产产物或阻塞证据
- 当用户要求高端、AAA、展示级或"不那么基础"的图形时，提供改动前后的视觉评分卡

当可用时，使用脚手架的 `npm run verify:visual` 和 `npm run inspect:canvas`，或使用本仓库的 `skills/threejs-qa-release/scripts/inspect-threejs-canvas.mjs`。
